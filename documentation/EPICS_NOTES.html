Introduction

This IOC application uses a LEEP driver to communicate with Osprey Event Generator and Event Receiver node.  The records mentioned in the following sections have names beginning with a $(P) macro.

Build Information

The time at which the firmware and software were built is contained in the per-node stringin records Date:FW-I and Date:SW-I, respectively.

Event Generator and Event Receiver Records

Event generator and event receiver records are described in the documentation for their respective support modules.

The MARBLE:MGTCLK0 mbbo record selects the multi-gigabit transceivers reference clock source.  Typical values are FPGA_REFCLK0 which selects the Marble system 125 MHz clock and FMC1_GBTCLK0 which selects the front panel SMA clock.  The value is stored in Marble non-volatile memory and used for subsequent FPGA reboots.

Machine Protection

The following table describes the records associated with the machine protection operations handled locally by each node.  The value o is the machine protection mitigation output number, ranging from 1 to 8.

An open or undriven input appears as a logic 0.   On power-up all outputs are in the 'tripped' state.  On the event generator node the outputs are driven by the combination of local tripped status from all nodes as described below by the MPS:required and  MPS:tripped records.  On all other nodes the outputs are driven by the local 'tripped' as described below by the MPS:status:o records.

Machine protection trips are cleared upon the arrival of event number 120.


Name
 Type
 Description
MPS:invert longout
 Bitmap specifying the local MPS hardware inputs to invert.  Intended primarily for MPS firmware/hardware validation.
MPS:chkInputs:o longout
 Least significant eight bits are bitmap specifying the local MPS hardware inputs for which a 'bad' value will trip the specified MPS output.
MPS:goodState:o longout
 Bitmap specifying the logic state (1/0) for which each local MPS hardware input will be considered 'good'.
MPS:status:o longin
 Least significant bit is the local node 'tripped' status.  It is set when any of the conditions above are met and cleared by an event from the top-level node.  If any of the conditions above are met when the clear event arrives the 'tripped' status remains set and the 'first fault' values are updated.
The next-to-least significant bit is set when any of the conditions that would cause a trip are present.
 Bits starting 16 and up are the current state of the local MPS hardware inputs after application of MPS:invert.
MPS:firstSeconds:o longin
 The time (POSIX seconds) at which 'tripped' status became set.
MPS:firstTicks:o longin
 The time (event system ticks) at which 'tripped status' became set.
MPS:firstInputs:o longin
 Bitmap of the local MPS hardware inputs that caused the 'tripped' status to become set.
MPS:forceTrip mbboDirect
 Bitmap of MPS outputs to force into the 'tripped' state.

In addition to the records above, event fanout and event generator nodes provide the following records to control the merging of MPS 'tripped' status from downstream nodes.

Name
 Type
 Description
MPS:required mbboDirect
 Bitmap specifying the fiber links from which MPS 'tripped' status will be merged.
The least significant bit is ignored since it corresponds to the incoming event receiver link.
MPS:tripped longin
 Least significant bits are map indicating the combined MPS 'tripped' status from all downstream nodes.
For the event generator the value of MPS:forceTrip is also combined and the value of MPS:tripped is the value written to the MPS mitigation hardware outputs.
 If a required fiber link is not receiving a valid stream all the bits in this record will be set.


Development Records

These records are intended for use during development.

Name
 Type
 Description
MARBLE:PPS:Local mbbo
0 Hardware PPS reference is obtained from RF-IN FMC card.
1
 System PPS reference is obtained from Marble system clock.  In this case the Marble clock synchronization PLL must be disabled.
2
 VCXO20 PPS reference is obtained from Marble 20 MHz VCXO (Y3).  The 20 MHz VCXO has a much wider adjustment range than the system clock 25 MHz VCXO (Y1) so care must be taken when enabling the Marble clock synchronization PLL.

MARBLE:VCXO:DAC_Y1 longout
 Writng a value of -32768 closes the phase-locked loop driving the Marble 25 MHz VCXO (Y1) used to generate system clocks.  Writing any other value opens the loop and drives the VCXO DAC with the specified value.
MARBLE:VCXO:DAC_Y3 longout
 Drives the Marble 20 MHz VCXO (Y3) DAC with the specified value.

Internal Records

Records with names ending in an underscore are intended for internal use by the IOC and should not be written by any other clients.
